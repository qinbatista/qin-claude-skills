# Adaptive And Admitted Routing Matrix

Eligible ordinary text/code production enters the compact adaptive runner exactly once, executes the saved contextual quality pair, and does not load the full `task-analyze-skill` unless routing strategy, benchmarking, maintenance, or a graph is in scope. Tool-only, image/mixed, and exact one-source read-only work stay inline. Two or three independent read-only sources first cost-admit one contextual producer versus a graph. `workflow-skill` executes only a positively admitted multi-node or locked route. Verification belongs to a separate post-result Ending task.

| Scenario | Use when | Foreground route | Model pattern | First presentation | Ending Task |
|---|---|---|---|---|---|
| ordinary-production | One eligible text/code result, edit, or implementation. | entry bootstrap -> adaptive producer -> result | Saved contextual quality pair; one stronger fallback only after zero result. | For code, run one bounded Quick Check; then present. | Create `End Task-{related task}` background Agent and return without waiting. |

## Goal Rules

- Every eligible ordinary text/code production task enters the adaptive runner once; exact one-source, tool-only, and image/mixed work remain inline.
- One receipt-valid Real PASS retains the selected pair, two matched PASS outcomes try one lower rung, and quality failure upgrades one rung.
- Priority producer is schedule-source-only and never overrides the contextual quality pair.
- Fan-out requires pre-read cost admission; dependent multi-file work always stays with one producer.
- Full Task Analyze activates only for explicit routing/benchmark/maintenance or graph evaluation; activation alone does not admit an open-ended graph.
- Every active registry-owned code-domain implementation uses `code-skill`.
- Main Result and proportional Quick Check precede all Ending verification.
- The origin creates and links a persistent `End Task-{related task}` background Agent, then returns without waiting, polling, repairing, or using a same-task subtask.
- Runtime receipts prove observed routing; labels and availability probes do not.
- Direct uses the exact raw prompt through `--direct-task`; Auto uses it through `--bootstrap-task`.
- The public two-world comparison is Direct task with zero verifier versus Auto task plus separate Ending. The common entry/controller dispatcher is disclosed but excluded from those task/check worlds.
- Savings require complete like-for-like evidence per task class. Logical tokens are not billing tokens.
- Do not publish or push unless explicitly requested.
